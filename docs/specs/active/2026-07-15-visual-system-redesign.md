# Whisper 视觉系统重建设计

日期：2026-07-15

## 背景

当前界面已经承载书库、阅读器、文本选择、设置和书籍 AI 会话，但视觉表现主要服务于功能验证，尚未形成与“原书是主要阅读对象，AI 是持续阅读伴侣”一致的产品语言。

本次改版彻底放弃现有视觉设计，从零建立 Whisper 的浅色视觉系统。现有信息架构、交互流程、页面状态、IPC、数据模型和主进程行为继续作为功能基线；这些内容的产品优化留给后续独立需求。

## 目标

- 建立具有清晰辨识度、适合长时间阅读的浅色视觉系统。
- 让原书始终成为阅读页的绝对视觉主角，AI 默认退居辅助位置。
- 统一书库、阅读器、AI 面板、设置、选区菜单及各种反馈状态。
- 使用 Base UI 承担复杂组件行为和无障碍基础，使用 CSS Modules 与 CSS Variables 完成视觉表现。
- 保留当前 renderer 模块边界、用户交互语义和跨进程数据流。

## 非目标

- 不优化书库、阅读器和 AI 会话的信息架构或交互流程。
- 不修改 IPC 契约、数据库、主进程服务或 preload API。
- 不实现深色模式，也不建立复杂的双主题切换机制。
- 不使用 Tailwind 或 shadcn/ui。
- 不引入在线字体或随应用打包新的字体文件。
- 不以本次改版为由提前建设完整的通用组件库。
- 不增加当前产品没有的新功能。

## 已确认的视觉方向

视觉隐喻为“学者的工作桌”。它强调原书、划线、批注、思考卡片和真实的阅读痕迹，但不采用仿木纹、纸张噪点、翻页动画等复古拟物装饰。

三个核心决策：

1. **学者的工作桌**：页面有书籍与研究材料的温度，AI 像书旁批注，而不是独立聊天产品。
2. **原书绝对主角**：中间阅读面最亮、最安静；目录和 AI 区域降低对比度，等待用户主动使用。
3. **琥珀与胡桃木**：旧纸作为阅读底色，深胡桃木作为结构色，琥珀只用于选区、批注和关键反馈。

视觉设计不得退化成通用 SaaS Dashboard、复古电子书或大面积棕黄色滤镜。

## 页面设计

### 阅读器

保留左侧目录、中间原书、右侧 AI 的三栏结构。

- 中间原书呈现为工作桌上最亮的一张纸，正文使用系统衬线字体栈、舒适行距和受控行宽。
- 左侧目录使用较暗且低对比度的工作台背景。当前章节清晰可辨，但不以高饱和色抢夺注意力。
- 右侧 AI 默认弱化，像尚未使用的研究笔记。用户选择文字、发问、切换到会话或收到流式回答时，内容区域获得更明确的层级。
- 文本选区使用克制的琥珀色标记；选区操作菜单保持紧凑，不遮挡大段原文。
- 会话标签、历史、解读目标、引用、输入区和流式状态沿用当前语义，只重建视觉层级。

### 书库

书库呈现为摆放书籍的工作台，而不是 Dashboard 卡片墙。

- 书籍封面是每个条目的主要视觉对象。
- 标题、作者、最近阅读位置和进度作为次级信息。
- 真实封面缺失时使用克制的排版封面，不在本次增加封面生成或网络抓取能力。
- 导入入口保持当前功能，仅重新设计位置、层级与状态。

### 设置

设置区域采用“桌边抽屉”的轻量面板感，优先保证字段、校验、保存状态和敏感信息的清晰度，不增加装饰性内容。

## 视觉基础

### Tokens

使用 CSS Variables 定义单一浅色主题，至少覆盖：

- 画布、工作台、纸张、浮层和结构边界；
- 主文本、次级文本、弱文本和反色文本；
- 胡桃木结构色、琥珀强调色和低饱和错误色；
- 间距、圆角、边框、阴影、层级和动效时长；
- 阅读正文宽度、字号、行高和段间距。

Token 名称表达语义而非具体颜色，例如 `--color-surface-paper`，不使用 `--beige-100` 作为业务层接口。

### 字体

- 阅读正文和书籍标题使用系统衬线字体栈，并为中文系统提供宋体类回退。
- 控件、导航、状态和输入使用系统无衬线字体栈。
- 代码、模型名和必要的技术信息使用系统等宽字体栈。
- 不发起字体网络请求，不打包第三方字体。

### 动效与材质

- 动效只用于解释状态变化、浮层出现和面板切换。
- 支持 `prefers-reduced-motion`。
- 纸张感依靠颜色、边界和克制阴影建立，不使用高频纹理影响正文清晰度。

## 组件与依赖策略

新增 `@base-ui/react`。Base UI 只负责复杂交互组件的行为和无障碍基础，例如 Dialog、Menu、Popover、Tooltip、Tabs 和 Field；它不定义 Whisper 的视觉语言。

组件策略遵守现有 renderer 架构：

- 页面继续位于 `src/renderer/pages/`，feature 继续位于 `src/renderer/features/`。
- 首次出现的视觉结构保留在所属 page 或 feature 内，不因为“可能复用”提前提升。
- 普通按钮、布局容器和书籍展示不为使用 Base UI 而强行封装。
- 只有被多个无关 feature 实际复用的 token、样式或组件才提升到 renderer 共享 UI 位置；不得重新创建已被 Harness 禁止的 `src/renderer/components/`。
- 业务模块不得绕过现有 renderer → preload API → IPC → main 的依赖方向。

## 状态设计

- **加载**：使用与最终布局一致的安静骨架或占位，避免布局跳变。
- **空状态**：说明当前为何为空，并提供现有能力中的下一步操作。
- **错误**：使用低饱和砖红色和明确文本；页面级错误保留恢复入口，局部错误留在对应区域。
- **禁用**：降低对比度，同时保留可读性和明确的不可操作状态。
- **焦点**：所有键盘可操作元素具有清晰、统一且不被阴影遮挡的 focus ring。
- **流式回答**：延续当前即时可见行为，通过排版和轻量状态提示区分思考、生成、完成和失败。

错误和加载表现不得改变现有命令执行、重试、会话恢复或错误归属逻辑。

## 实施顺序

采用“阅读器样板 → 提炼系统 → 覆盖其余页面”的方式：

1. 建立浅色 tokens、系统字体栈和必要的 Base UI 基础。
2. 以阅读器为视觉样板，覆盖目录、原书、选区菜单和 AI 面板的主要状态。
3. 从已经验证的阅读器实现中提炼真实复用的样式或组件。
4. 重建书库与设置。
5. 补齐加载、空、错误、禁用、焦点、流式和窄窗口状态。
6. 完成自动检查与人工视觉验收。

现有页面不需要在中间阶段保持新旧视觉混用的发布质量，但提交完成时所有用户可达界面必须统一到新系统。

## 测试与验收

自动测试重点验证现有交互语义没有回归：

- 返回书库和打开书籍；
- 章节跳转；
- 文本选择及选区菜单操作；
- 新建会话、引用原文、切换 Tab 和历史恢复；
- 流式回答的思考、增量文本、完成和失败；
- 设置加载、修改、保存与错误反馈；
- Base UI 浮层的键盘操作、焦点管理与关闭行为。

人工验收至少覆盖：

- 常见 macOS 与 Windows 系统字体下的中英文混排；
- 长章节、长标题、无封面、空书库和多会话；
- 键盘导航、清晰 focus ring 和 reduced motion；
- 常用窗口尺寸及较窄窗口下无关键操作丢失；
- 正文长时间阅读时，目录与 AI 面板不会持续抢夺注意力；
- 所有用户可达页面均已移除旧视觉残留。

完成前运行 `pnpm check`。若引入 Storybook，则将组件关键状态作为可视化验收面；是否引入 Storybook 在实施计划中依据组件规模决定，不作为本规格的强制依赖。

## 验收条件

- 阅读器、书库、设置、AI 会话和选区界面形成一致的“学者工作桌”浅色视觉系统。
- 原书在阅读页始终是绝对视觉主角，AI 默认处于辅助层级。
- 不存在 Tailwind、shadcn/ui、在线字体或打包字体依赖。
- Base UI 仅承担所需行为，外观由 CSS Modules 与 tokens 控制。
- 当前信息架构、交互流程、IPC、数据模型和主进程行为没有未经说明的变化。
- 关键交互测试通过，`pnpm check` 通过，人工视觉验收完成。
