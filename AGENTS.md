# Whisper 项目地图

## 语言

面向用户的沟通、规格、计划、评审和说明默认使用中文；代码标识符、API、命令、路径和错误原文保留英文。

## 产品

Whisper 是本地优先的 Electron 阅读伴侣：原书是主要阅读对象，AI 负责解释、补充背景和围绕书籍持续对话。

## 仓库地图

- `src/main/`：主进程、本地数据、书籍解析、AI 服务和 IPC handler。
- `src/preload/`：最小化的 renderer IPC 桥。
- `src/renderer/`：React 界面与页面状态。
- `src/shared/`：跨进程契约和纯领域定义。
- `tests/`：与源码分层对应的 Vitest 回归测试。
- `ARCHITECTURE.md`：当前进程边界、模块职责和关键数据流。
- `docs/`：规格、计划、架构决策、运维、质量与人工验收资料。

## 变更与文档路由

代码、测试、类型和提交记录本身也是仓库记录。文档投入必须与复杂度匹配；优先更新已有权威文档，不为同一主题重复建文件。

- 微小改动：直接修改，以代码、检查结果和提交记录作为记录。
- Bug 修复：优先增加回归测试；只有产生新不变量、兼容性或安全约束时才更新文档。
- 小需求：明确验收条件后直接实现；局部且清楚的行为变化不要求独立 Spec。
- 中大型需求：跨多个进程或模块、修改 IPC 或数据模型、存在重要方案取舍时，在 `docs/specs/active/` 创建或更新 Spec。
- 多阶段、需要跨会话实施或需要交接的任务，在 `docs/plans/active/` 创建 Plan。
- 长期架构或技术取舍记录到 `docs/decisions/`，形成 ADR。
- 只有需求模糊、影响广或实施复杂时才使用 Superpowers；创建 Spec/Plan 时必须显式指定上述 active 路径，不使用工具默认目录。
- 需求边界不清或重要假设未验证时，可在实现前使用 `grill-me` 做追问式压力测试，或使用 `superpowers:brainstorming` 梳理目标、非目标和方案取舍；明确 Bug、微小改动和边界清楚的小需求不强制使用。
- 可重复的运行、排障、迁移和恢复流程记录到 `docs/operations/`。
- 系统性质量风险和治理重点记录到 `docs/quality/`；自动测试覆盖不了的关键用户旅程加入 `docs/MANUAL_TESTING.md`。
- `README.md` 只面向人类介绍产品、开发入口和导航；`AGENTS.md` 不承载具体功能设计。
- 完成的 Spec/Plan 移入 `completed/`；被替代的 Spec 移入 `superseded/` 并注明 `supersededBy`。

## 架构不变量

- 依赖方向为 `renderer -> preload API -> IPC -> main`；renderer 不得导入 main、Electron 或 Node API。
- `shared` 不依赖 Electron、React 或具体业务实现；preload 不透传任意 IPC 能力。
- IPC、文件内容和模型响应等外部输入必须在主进程边界做运行时校验。
- SQLite 只由主进程访问；schema 变更必须有数据兼容策略和数据库回归测试。
- 日志不得泄露 API key 等敏感信息；新增敏感字段时同步扩展脱敏规则。
- 不得以删除用户数据库代替正式迁移，除非 Spec 明确限定为开发期破坏性变更。

## 标准工作流

1. 先判断变更级别，按需选择直接修改、Spec、Plan 或 ADR，不默认创建文档。
2. 阅读邻近代码和测试，并按需读取 `ARCHITECTURE.md` 及相关 active 文档。
3. 修改代码并补充与风险相称的测试；可机械验证的约束优先写成测试或静态检查。
4. 完成前运行 `pnpm check`；涉及真实交互时执行相关人工验收项。
5. 中大型需求、连续多项小改动、同类 Bug 再现或发布前，回看受影响模块的职责、重复模式和临时兼容逻辑，必要时立即重构或升级机械约束。
6. 只有产生长期有效的信息时才更新文档，并同步其生命周期状态。

## 完成定义

- 验收条件已满足，没有未说明的行为变化或剩余风险。
- 相关测试已新增或更新，`pnpm check` 通过，既有不变量未被破坏。
- 必要的产品说明、架构决策、操作方法和人工验收结果已同步到正确位置。
